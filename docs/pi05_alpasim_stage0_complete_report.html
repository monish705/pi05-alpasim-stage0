Abstract

We adapted a PI0.5 vision-language-action model, originally configured for robot-action generation, into a custom external driver for NVIDIA AlpaSim. The Stage 0 goal was narrow: verify that a lightly fine-tuned PI0.5 checkpoint can run in closed loop, emit car-compatible actions through a kinematic bridge, and control a vehicle in the simulator on a same-scene route. The baseline same-scene run completed successfully and produced valid video, trace, and metric artifacts. We then ran a four-condition camera ablation with raw per-call dumps: A all cameras live, B front camera only, C all cameras black, and D wrong-scene static override images. The combined evidence shows that the policy is genuinely running and influencing control, but that the current stack is not yet visually grounded enough to support a driving-performance claim.

System Setup

Model and Training

  • Base model: pi0.5
  • Fine-tune mode: LoRA on the PI0.5 JAX stack
  • Checkpoint used for eval: /mnt/data/checkpoints/pi05_stage0_av/pi05_stage0_av/1000
  • Native action tensor preserved: 50 x 32
  • Active dims: delta_s, delta_yaw, target_speed
  • Dataset scale: 5 episodes, 545 total frames, 10 Hz

Simulator Runtime

  • AlpaSim external-driver mode on localhost:6789
  • Three logical cameras: camera_front_wide_120fov, camera_cross_left_120fov, camera_cross_right_120fov
  • Control timestep: 100 ms
  • Bridge clamps: acceleration, speed, yaw-rate, lateral acceleration
  • Same-scene eval scene: clipgt-048b974e-1546-488a-b8f9-d32bff77f5aa

Main Findings

The fine-tuned PI0.5 model is genuinely running in the loop.
Every ablation produced 101 policy calls, nontrivial output variation, valid trace.jsonl, and a completed rollout directory with metrics.parquet, rollout.asl, and MP4.
The current result is not a vision-grounded driving-performance result.
Condition A is not stronger than C or D on the key route-progress metric. In fact, all-black C and wrong-scene D both exceed A on progress_last. That means the current stack is still too route/bridge/controller dominated to claim robust perception-grounded driving.
Vision is affecting the policy, but not in the way you ultimately need.
The clearest signal is speed regulation. All-black C raises target_speed_first_mean to 8.589554 m/s and doubles accel-clamp pressure to 1028. So the visual pathway is not dead. It appears to modulate speed/confidence more than it provides reliable turn-disambiguation.
The bridge is still rescuing the model heavily.
Acceleration clamp totals remain high in every condition: A 559, B 543, C 1028, D 715. That means the policy often emits speed profiles that need feasibility correction before the controller can track them.
Latency remains too high for a clean 10 Hz closed loop.
Warm end-to-end latency is around 243–250 ms, while the control timestep is 100 ms. The model may be reasonable, but the controller is still acting on stale information.
The offroad metric is still contaminated.
Offroad remains 1.0 across all conditions. That means the spawn/eval issue is still unresolved and you should not use offroad as your primary comparative metric yet.
Condition C is prompt-confounded.
C includes left_turn in its prompt set while A, B, and D only show lane_follow and right_turn. That makes yaw comparisons with C less clean than the speed/clamp comparisons.

Claim Boundary

What you can honestly claim now

  • PI0.5 can be fine-tuned and run as a custom AlpaSim external driver.
  • The model can execute repeated closed-loop inference and complete simulator rollouts.
  • Raw per-call VLA inputs and outputs can be captured and inspected.
  • The current stack exposes a clear next bottleneck: vision grounding.

What you cannot claim yet

  • Strong driving performance.
  • Vision-grounded route following.
  • Generalization beyond this narrow scene family.
  • Superiority over a driving-native baseline.

Recommended Next Step

  1. Fix the spawn/offroad evaluation contamination so offroad becomes usable.
  2. Repeat A/B/C/D with identical prompt sets, especially for C.
  3. Run at least 5 rollouts per condition to report mean and standard deviation.
  4. Reduce route dominance and bridge dependence before making any performance claim.
  5. Add a held-out same-domain scene and a simple driving-native baseline.
