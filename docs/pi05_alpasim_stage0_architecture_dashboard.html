What the Model Sees and Predicts

Visual Inputs

  • camera_front_wide_120fov
  • camera_cross_left_120fov
  • camera_cross_right_120fov
  • Driver remaps them into OpenPI image slots: base_0_rgb, left_wrist_0_rgb, right_wrist_0_rgb

Structured Inputs

  • 1.0 s ego history at 10 Hz
  • Per-step features: speed, yaw rate, acceleration
  • State history flattened to 30 dims
  • Route corridor: 32 x 2 future ego-frame points
  • Prompt token: lane_follow, left_turn, or right_turn

Action Semantics

  • Internal tensor preserved at 50 x 32
  • Only active dims: 0 = delta_s, 1 = delta_yaw, 2 = target_speed
  • Other 29 dims are present structurally but ignored semantically by this Stage 0 bridge

Training Pipeline Actually Used

1

Scene Selection

The selector validates NuRec labels, requires at least one explicit left turn and one right turn, then picks exactly 5 clips: left, right, and 3 lane-follow clips.

select_stage0_candidates.py

2

Dataset Build

Raw AV cameras and egomotion are aligned to a 10 Hz timeline, RGB frames are decoded, ego kinematics are derived, and route/action targets are computed.

build_stage0_dataset.py

3

Normalization

State and action stats are computed from the Stage 0 dataset itself. The script validates non-finite values, zero std, and bad quantiles before saving assets.

compute_stage0_norm_stats.py

4

OpenPI JAX Fine-Tune

A Stage 0 wrapper constructs the OpenPI training config, repacks observation tensors, exposes only the active action dims on output, and calls the upstream JAX train script.

openpi_stage0.py

5

Runtime Bridge + Driver

At inference time, normalized outputs are denormalized, rolled into a feasible planar trajectory under kinematic limits, then served to AlpaSim via the custom external driver.

bridge.py / pi05_stage0_model.py

Loss / Reward / Objective: What Was Actually Used

What we did use

  • OpenPI JAX training path
  • LoRA fine-tuning on the PI0.5 stack
  • Stage 0 data transforms and normalization
  • Checkpoint weight initialization from PI0.5 base params

What we did not add

  • No RL reward function in this workspace
  • No PPO / policy-gradient loop
  • No learned simulator reward model
  • No custom loss file overriding the upstream OpenPI train objective

What acts like "safety"

  • Inference-time feasibility bridge
  • Acceleration clamps
  • Yaw-rate clamps
  • Lateral-acceleration clamps
  • These are post-model corrections, not training rewards

Why this matters

The model does not directly hand AlpaSim a finished physically feasible car trajectory. The bridge takes the predicted action chunk, denormalizes it, and rolls it forward while enforcing speed, acceleration, yaw-rate, and lateral-acceleration limits.

Implication

Reported closed-loop behavior is the result of model + bridge + controller, not raw unconstrained model output alone.
